# jiaojie

## 目标

用最小但可靠的交接信息稳定跨线程续做：先读交接，再按需扩展，再校验后交付。

## 先做路由

1. 判断模式：
   - **收尾交接**：当前批次接近结束，或用户要求“更新交接包 / 给下个线程交接”
   - **继续交接**：新线程刚开始，或用户要求“按交接继续 / 先读交接文件继续”
   - **重建交接**：交接缺失、过期严重、明显空洞、验证失败、或旧结论已不可信
2. 默认交接文件：
   - `docs/codex/handoff.md`
   - `docs/codex/current-phase.md`
   - `docs/codex/next-thread-prompt.md`
   如果仓库已有等价文档且位置更合理，优先复用，不平行新建。
3. 始终先读交接三件套，再按交接提示补读主文档、关键配置、最近改动与必要代码。除非交接失效，不要上来全仓摸底。

## 核心工作流

1. 读现有 handoff 三件套；必要时再读主文档、最近改动、验证脚本结果。
2. 只保留对下一线程有行动价值的信息：目标、已完成、风险、下一步、限制、可信来源。
3. 更新或重建交接文档；如本轮确认主文档健康状态有变化，同步修正交接里的陈旧表述。
4. 在宣称“交接完成 / 文档损坏 / 可继续”前，先跑必要校验。
5. 回复用户时至少说明：模式、读取/更新文件、当前状态一句话、下一步建议。

## 不可违背的规则

- **轻量优先**：稳定事实进 `handoff.md`；当前批次进 `current-phase.md`；`next-thread-prompt.md` 只保留最小启动信息。
- **先证伪，再判坏**：不要只因终端乱码就判定文档损坏；至少先做显式 UTF-8 读取与脚本校验。
- **状态一致**：如果主文档已修复/已健康，必须在同一轮撤销 handoff 中“已损坏 / 需重建 / 不可信”的旧状态。
- **Windows 写入安全**：优先 `apply_patch`；只在正文来自本地模板、已有文件或短字符串时用 `Path(...).write_text(..., encoding='utf-8')`；避免 here-string、stdin 管道直落长中文正文。
- **不要让用户重复总结**：继续交接时先读现有交接，再直接续做。

## 配套脚本

- `scripts/validate_handoff.py`：检查交接/主文档结构、编码与明显损坏迹象
- `scripts/audit_doc_health.py`：检查 handoff 是否仍保留对“已健康主文档”的陈旧负面表述
- `scripts/diagnose_encoding.py`：诊断本机 Python / 终端编码链路

## 默认理解

以下表达默认按本技能处理：

- `收尾交接`
- `继续交接`
- `重建交接`
- `交接包`
- `jiaojie`
- `$jiaojie`

若用户只说 `jiaojie`，按上下文判断：

- 当前批次接近结束：优先收尾交接
- 新线程刚开始且交接可用：优先继续交接
- 交接缺失、过期或损坏：优先重建交接

## 一句话

`jiaojie` 的核心不是多写文档，而是用最小、可信、可校验的交接，稳定续做并避免在 Windows 中文环境下把文档悄悄写坏。
