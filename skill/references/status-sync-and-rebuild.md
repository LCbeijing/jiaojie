# 状态同步与重建写法

何时读：本轮改变了主文档健康结论，或需要重建 handoff 并决定怎么写“事实 / 推断 / 旧结论过期”时。

## 健康状态只有三类

### 1. 当前健康

适用：主文档当前可读、结构校验通过、可继续作为可信来源。

推荐写法：

- `当前健康`
- `已通过校验`
- `可作为可信来源`

### 2. 旧结论过期

适用：历史 handoff 曾写“已损坏 / 需重建”，但当前校验已证明文档可用。

推荐写法：

- `旧结论已过期`
- `当前文件可按 UTF-8 正常读取并通过校验`
- `应撤销历史“已损坏”描述`

### 3. 当前确实不健康

适用：当前显式读取失败、结构校验失败、或有明显损坏迹象。

推荐写法：

- `当前校验失败`
- `暂不能作为可信来源`
- `需重建后再作为主文档使用`

不要在没有当前校验依据时直接写“已损坏”。

## 主文档状态变化后的同步要求

如果本轮修复、重建或确认某个主文档已恢复健康，必须在**同一轮**同步更新：

- `docs/codex/handoff.md`
- `docs/codex/current-phase.md`
- `docs/codex/next-thread-prompt.md`

至少做到：

1. 删除旧的“已损坏 / 需重建 / 不可信”表述
2. 改成“当前健康 / 已通过校验 / 可作为可信来源”
3. 把下一步中的“先重建主文档”改成当前真实动作

## 重建 handoff 时的证据分层

明确区分：

- **已确认事实**：来自当前文件、当前代码、当前校验结果
- **推断项**：来自仓库结构、最近改动、线程上下文的合理反推

写法要求：

- 事实可以直接陈述
- 推断要显式标注，不要写成已证实结论

## 建议保留 / 建议删除

建议保留：

- `已重建并通过校验`
- `当前健康`
- `可作为可信来源`
- `状态一致`
- `旧结论已过期`

建议删除或改写：

- `已损坏`
- `真实损坏`
- `需后续重建`
- `不要作为首要可信来源`
- `不能继续作为高可信主文档`

## 信息来源优先级

1. 当前可读的 handoff 三件套
2. 当前主文档
3. 最近改动与关键文件
4. 当前线程上下文
5. 仓库结构与运行证据

优先使用**当前可验证来源**，而不是沿用历史猜测。
