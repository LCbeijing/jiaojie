# 模式手册

何时读：判断 `jiaojie` 该走哪条链路，或执行中途需要切模式时。

目录：
- 一眼路由
- 收尾交接
- 继续交接
- 重建交接

## 一眼路由

- **收尾交接**：当前批次接近结束；目标是给下线程留下最小可续做信息。
- **继续交接**：新线程刚开始且现有 handoff 可用；目标是先读交接再直接续做。
- **重建交接**：handoff 缺失、空洞、严重过期、校验失败、或与当前仓库明显冲突；目标是重新建立可信 handoff。

默认优先级：用户明确指定 > 用户只说 `jiaojie` 时按上下文判断 > 执行中一旦发现 handoff 不可信立即切重建。

## 收尾交接

目标：把本轮真正影响下线程的信息压缩进 handoff 三件套，不把当前线程的临时噪音带过去。

最小读取集：
1. handoff 三件套
2. 本轮最近改动文件 / 最近验证结果
3. 只有当 handoff 提示主文档状态变化时，才补读主文档

执行：
1. 提炼本轮的目标、完成状态、风险、下一步、限制
2. 分发到三件套：稳定事实 / 当前批次 / 下一线程启动动作
3. 跑基础校验；若主文档健康状态变化，再跑一致性审计
4. 用一句话概括“现在到哪了，下一步该干什么”

完成标准：
- 三件套已更新或确认无需改动
- 没有长日志、调试聊天、重复背景
- 下线程无需重新全仓摸底

常见误区：
- 把完整调试过程塞进 `handoff.md`
- 把“下一步动作”写得过空
- 主文档已恢复健康，却仍保留“已损坏 / 需重建”旧结论

## 继续交接

目标：把 handoff 当成入口，而不是让用户重复口头总结项目状态。

最小读取集：
1. handoff 三件套
2. handoff 指向的主文档 / 关键配置 / 最近改动
3. 必要时再看验证脚本结果

执行：
1. 先读三件套并用中文简要复述项目目标、当前阶段、下一步
2. 只按交接提示补读必要材料，不全仓重摸
3. 若 handoff 提到“主文档损坏 / 需重建”，先验证原文件再沿用结论
4. 直接续做，不要求用户重复总结历史

何时切到重建：
- handoff 缺失
- handoff 明显过期或空洞
- handoff 与当前仓库状态冲突
- handoff 声称主文档损坏，但拿不出当前校验依据
- 校验脚本失败，或显式读取后确认内容不可用

## 重建交接

目标：在不可信 handoff 之上重建最小但可靠的交接基线。

最小读取集：
1. 现存 handoff（如果还有）
2. 主文档
3. 最近改动与关键文件
4. 当前线程上下文

执行：
1. 从当前可验证来源反推项目状态
2. 重新生成三件套
3. 明确标注“已确认事实”与“推断项”
4. 跑基础校验；如涉及健康状态变化，再跑一致性审计
5. 撤销陈旧“已损坏 / 需重建”描述

完成标准：
- 新 handoff 足以支撑新线程继续开发
- 事实与推断没有混写
- handoff 与主文档状态一致

常见误区：
- 把历史猜测继续写成当前事实
- 用全仓摸底替代“按当前可信证据重建”
- 主文档已健康却没有同步回写 handoff
