# 交接文件契约

何时读：判断“这条信息该写去哪一份文档”，或需要重写三件套时。

目录：
- 先分流再落笔
- `handoff.md`
- `current-phase.md`
- `next-thread-prompt.md`
- 等价文档复用 / 常见误放

## 先分流再落笔

- **跨阶段稳定事实** → `handoff.md`
- **当前批次状态** → `current-phase.md`
- **给下线程的直接启动动作** → `next-thread-prompt.md`

如果一条信息不能帮助下线程更快继续，就不要写进 handoff 三件套。

## `handoff.md`

用途：长期总账；只保留跨阶段稳定事实，不写流水账。

必须有：项目名称 / 仓库路径、当前项目目标、可编辑基线 / 运行证据基线、已完成核心成果、固定约束与冻结规则、已知风险、下一阶段建议、可信来源 / 验证依据、最后更新时间。

最小结构：

```markdown
# 项目交接总账
## 1. 项目目标
## 2. 基线定义
## 3. 已完成核心成果
## 4. 固定约束与冻结规则
## 5. 已知风险与遗留问题
## 6. 下一阶段建议
## 7. 可信来源 / 验证依据
## 8. 最后更新时间
```

应强调：哪些事实已经稳定、哪些来源当前可信、下一阶段大方向。
不要放：长工具输出、临时调试细节、与下线程无关的试错过程、纯历史背景复述。
合格标准：下线程读完后，能知道“项目现在依靠哪些可信事实继续往前走”。

## `current-phase.md`

用途：当前阶段执行板；比 `handoff.md` 更短、更聚焦、可高频更新。

必须有：当前阶段目标、本批已完成事项、最近改动文件 / 模块、当前验证结果、当前阻塞 / 风险、下一步最优先动作、最后更新时间。

最小结构：

```markdown
# 当前阶段进展
## 1. 当前阶段目标
## 2. 本批已完成事项
## 3. 最近改动文件 / 模块
## 4. 当前验证结果
## 5. 当前阻塞 / 风险
## 6. 下一步最优先动作
## 7. 最后更新时间
```

应强调：本批已经做完什么、现在卡在哪、下一步立刻做什么。
不要放：长期背景、与当前阶段无关的历史决策、可直接从 `handoff.md` 引用的稳定信息。
合格标准：下线程读完后，能直接知道“现在先干什么”。

## `next-thread-prompt.md`

用途：给下一线程直接复制使用；短、准、可执行。

必须有：项目路径、先读哪些文件、当前阶段目标、不要重新摸底、直接开始的动作、关键限制。

最小结构：

```markdown
你在仓库 `<repo>` 中继续工作。

先读取以下文件：
- `<handoff>`
- `<current-phase>`
- `<main-docs-if-needed>`

当前阶段目标：
- …

不要重新全仓摸底；先按交接继续。

直接开始：
1. …
2. …

关键限制：
- …
```

应强调：先读什么、现在做什么、哪些限制必须继续遵守。
不要放：详细背景解释、长段验证日志、不影响下一步动作的历史争议。
合格标准：下一线程复制后即可开工，不需要再问“那我先看什么、先做什么”。

## 等价文档复用

如果仓库已有等价交接文件：
- 优先复用，不平行新建
- 仍按上面的职责分层写
- 文件名不是标准名时，用 `::kind` 显式指定类型（见 `references/validation-and-health.md`）

## 常见误放

- 把“本批验证结果”写进 `handoff.md`
- 把“长期约束”只写进 `current-phase.md`
- 把 `next-thread-prompt.md` 写成背景介绍而不是启动指令
